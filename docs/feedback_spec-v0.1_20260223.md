# URL/PDFナレッジ投入型 AIチャットボットSaaS — 開発上の懸念点と改善アドバイス（v0.1付随）

本ドキュメントは、初期仕様書（v0.1）に基づき、開発・事業化の過程で直面する可能性が高い技術的課題および機能拡張の優先順位を整理したものである。

---

## 1. 技術的懸念：ナレッジ取得の網羅性
仕様書では「URL投入」を静的HTMLの取得（HTTP fetch）としているが、現代のWebサイトの多くは動的な描画を伴う。

- **課題**: クライアントサイドでレンダリングされるサイト（SPA等）の場合、単純なfetchでは本文が抽出できないリスクがある。
- **アドバイス**: 
  - Phase 1以降で予定されているHeadlessブラウザ（Playwright/Puppeteer等）の導入を、主要なニーズに合わせて前倒し検討する。
  - 1 URL単位だけでなく、サイトマップ（sitemap.xml）を読み込んでドメイン配下を自動巡回する機能の優先度を上げる。

## 2. 信頼性の担保：出典表示（Citations）の扱い
仕様書では、参照元の表示を「プランによるON/OFF」としているが、RAGにおいて出典はハルシネーション（嘘の回答）を防ぐための最重要機能である。

- **課題**: 出典が不明な回答は、ビジネス用途（社内規程や製品マニュアル）では信頼を損なう原因となる。
- **アドバイス**: 
  - 出典表示は全プランで標準搭載とし、プラン間の差は「詳細度（PDFのページ数表示やリンクの有無など）」でつけることを検討する。

## 3. 日本語特有の検索精度（チャンク化の最適化）
仕様書では、上限トークン（500〜1,000）での分割を基本ルールとしている。

- **課題**: 日本語はトークン分割の際に文の中途で切断されやすく、ベクトル検索の精度（意味の保持）が低下する恐れがある。
- **アドバイス**: 
  - トークン数だけで機械的に切るのではなく、「句点（。）」や「改行」などの論理的な区切りを優先しつつ、オーバーラップを調整する日本語最適化ロジックを採用する。

## 4. セキュリティと機密情報の取り扱い
「社内規程」や「業務マニュアル」を対象とする以上、情報の機密保持は極めて重要である。

- **課題**: 顧客は「自分たちが投入したデータがLLM（OpenAI等）の学習に再利用されないか」を非常に強く懸念する。
- **アドバイス**: 
  - API経由でのデータ利用は学習に利用されないという規約上の事実を、LPや管理画面で明文化し、B2Bにおける安心感を醸成する。
  - 第10.4項のログ保持に関連し、個人情報（PII）の自動マスキング機能の検討をロードマップに加える。

## 5. 運用性の向上：フィードバック・ループの構築
MVPでは「回答の生成」までが主眼となっているが、実運用では「回答の質をどう改善するか」が重要になる。

- **課題**: 顧客（管理者）側で「どのナレッジが不足していて回答できなかったか」を知る手段が必要である。
- **アドバイス**: 
  - チャットUIに「Good/Bad」ボタンを設置し、管理画面で低評価の質問を一覧化できる機能を早期に追加する。
  - これにより、顧客が自発的にナレッジ（Source）を補充・修正する運用サイクルが生まれる。
