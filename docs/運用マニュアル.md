## 1. 管理コンソール概要

### アクセス方法

| 項目 | 内容 |
|------|------|
| URL | `https://operations.knotic.<ドメイン>` |
| ログインページ | `/sub-domain/login` |
| 認証方式 | メールアドレス + パスワード（Supabase Auth） |

> **注意:** 通常の契約者コンソール（`/console`）とはセッションが独立しています。
> 同じブラウザで両方にログインしていても、管理コンソールへのアクセスには別途ログインが必要です。

### 管理者権限の種類

| ロール | 説明 |
|--------|------|
| `owner` | 全操作が可能。プランリミット変更・Override無効化など破壊的操作を含む。 |
| `staff` | テナント管理・契約Override設定・代理閲覧は可能。プランリミット変更は不可。 |

管理者アカウントは Supabase Dashboard の `platform_admin_users` テーブルで管理します。

### コンソール機能一覧

| メニュー | パス | 概要 |
|----------|------|------|
| ダッシュボード | `/sub-domain` | テナント一覧・検索・手動契約操作 |
| テナント作成 | `/sub-domain/tenants/new` | 新規テナントの作成 |
| 監査ログ | `/sub-domain/audit-logs` | 全操作ログの確認・フィルタ |
| インデックスジョブ | `/sub-domain/indexing-jobs` | RAGインデックス処理の状態監視 |
| プラン管理 | `/sub-domain/plans` | プランごとのリミット値編集（owner のみ） |
| 運用ドキュメント | `/sub-domain/docs` | コンソール内操作ガイド |

## 2. テナント管理

### テナントとは

Knotic における「テナント」は1つの契約単位です。1テナント = 1社（または1チーム）に対応します。
テナントの下に複数の Bot・ソース・メンバーが紐づきます。

### テナント一覧

**パス:** `/sub-domain`

- 作成日降順で最大 **30件/ページ** 表示
- slug または表示名（display_name）でテキスト検索可能
- Stripe 契約状況・手動 Override の両方を同時表示

**ステータスバッジの見方**

| バッジ | 意味 |
|--------|------|
| `有効`（緑） | active = true。通常稼働中。 |
| `無効` | active = false。解約済み扱い。 |
| `強制停止中`（赤） | force_stopped = true。チャット応答をブロック中。 |

### テナント詳細ページのタブ構成

**パス:** `/sub-domain/tenants/[tenant_id]`

| タブ | 主な操作 |
|------|----------|
| 概要 | テナントの有効化状態・強制停止の切り替え |
| 契約・請求 | Stripe契約確認・Override設定・Stripe決済URL発行 |
| メンバー | メンバー一覧・招待メール送信・権限付与 |
| Bot管理 | Bot一覧・個別Bot強制停止 |

### テナント新規作成

**パス:** `/sub-domain/tenants/new`

| 項目 | 必須 | 説明 |
|------|------|------|
| 表示名（display_name） | ✅ | テナントの表示名。顧客企業名など。 |
| スラッグ（slug） | 任意 | URLに使用する識別子。省略すると表示名から自動生成。 |
| オーナーメールアドレス | ✅ | **Supabase に登録済みのアカウント** のメールアドレスを指定。 |
| 請求形態を設定する | 任意 | ON にすると Override も同時作成。銀行振込・請求書払い向け。 |

> **注意:** オーナーメールアドレスが未登録の場合はエラーになります。
> 先に「メンバー」タブの招待メール送信でアカウントを作成してから、テナント作成を行ってください。

### テナントの有効化状態管理

**場所:** テナント詳細 → 概要タブ → 「テナント有効化状態」カード

| 状態 | 意味 |
|------|------|
| 有効（緑トグル） | 通常稼働中。 |
| 無効（グレートグル） | 解約済み扱い。コンソールログインは可能だが、プラン判定が無効扱いになる。 |

操作手順: トグルを切り替え → 「有効化する」または「無効化する」ボタン → 確認ダイアログで「確定する」

> ⚠️ **無効化しても Stripe サブスクリプションは自動キャンセルされません。Stripe Dashboard での操作も別途必要です。**

### 代理閲覧（インポーソネーション）

管理者が特定テナントの契約者コンソールを読み取り専用で閲覧できる機能です。顧客サポートや障害調査に使用します。

**開始手順:**

1. テナント詳細ページの「代理閲覧」ボタンをクリック
2. 確認ダイアログで「開始する」
3. `/console/overview` へ遷移し、画面上部に「代理閲覧中」バナーが表示される

**終了:** バナーの「終了」ボタンをクリック、または1時間経過で自動終了

> 🚨 **代理閲覧中は書き込み操作がすべてブロックされます（読み取り専用）。**
> 開始・終了は監査ログに記録されます（`platform.impersonation.start / stop`）。

### テナント削除について

テナント削除機能は**実装されていません**。解約・利用停止の場合は以下を組み合わせて対応してください。

| 操作 | 方法 |
|------|------|
| チャット応答を即時停止 | 強制停止（force_stop = true） |
| 解約済み扱いにする | 有効化状態を無効（active = false）に変更 |
| Stripe 解約 | Stripe Dashboard で別途対応 |

## 3. 契約・請求管理（Override）

### 契約管理の2系統

Knotic の契約管理は **Stripe** と **手動 Override** の2系統があります。

```
プラン判定の優先順位:
  Override（is_active = true）> Stripe subscription
```

Override が有効な場合、Stripe の契約状態は**完全に無視**されます。

### Stripe 契約の subscription status

| status | 意味 |
|--------|------|
| `trialing` | トライアル中。機能は利用可能。 |
| `active` | 有効な契約。正常稼働。 |
| `past_due` | 支払い遅延。催促中。機能は一時的に利用可能。 |
| `unpaid` | 支払い未完了。機能はブロックされる。 |
| `canceled` | 解約済み。機能はブロックされる。 |
| `paused` | 一時停止中。 |
| `incomplete` | 初回決済が未完了。 |

### 手動 Override の使いどころ

| billing_mode | 説明 |
|--------------|------|
| `bank_transfer` | 銀行振込払いの顧客。請求書発行後に手動で有効化する。 |
| `invoice` | 請求書払い（インボイス対応）の顧客。 |
| `manual` | 社内利用・無料提供・個別交渉プランなど。 |

### Override の設定手順

**場所:** テナント詳細 → 「契約・請求」タブ → 「手動オーバーライド 設定・更新」カード

| 項目 | 説明 |
|------|------|
| plan_id | 適用するプランを選択 |
| status | 契約ステータス（通常は `active`） |
| billing_mode | 請求方式を選択 |
| effective_from | 適用開始日時（ISO形式。例: `2026-04-01T00:00:00+09:00`） |
| effective_until | 適用終了日時（空欄の場合は無期限） |
| 備考 | 契約メモ・請求条件など内部メモ |
| is_active トグル | **有効** = 即時適用 / **無効** = 保存のみ（Stripe に戻る） |

### is_active トグルの意味

| 状態 | 動作 |
|------|------|
| **ON（有効・緑）** | このOverrideを即時適用。Stripeの契約状態は無視される。 |
| **OFF（無効・グレー）** | 設定はDBに残るが適用されない。Stripeに基づく判定に戻る。 |

> Override を一時的に止めてStripeに戻したい場合は、設定を削除せずに is_active を OFF にするだけでOKです。

### Stripe 決済URL の発行

**場所:** テナント詳細 → 「契約・請求」タブ → 「Stripe 決済URL発行」カード

顧客に Stripe Checkout の支払いページURLを共有する機能です。有効期限は発行から約24時間です。期限切れの場合は再発行してください。

## 4. メンバー・ユーザー管理

### メンバーの権限（ロール）

| ロール | 説明 |
|--------|------|
| `editor` | Bot作成・設定変更・ソース追加など全操作が可能。 |
| `reader` | コンソールの閲覧のみ。データ変更・Bot操作は不可。 |

### 招待メール送信（未登録ユーザー向け）

**場所:** テナント詳細 → 「メンバー」タブ → 「招待メール送信」フォーム

1. メールアドレスとロールを入力して「招待メールを送信」
2. Supabase が招待メールを自動送信
3. 受信者がリンクをクリックしてパスワードを設定するとアカウント作成完了
4. メンバーシップは招待時点で事前登録済みのため、受け入れ後すぐにコンソールへアクセスできる

> 同一メールアドレスへの重複招待は「既存アカウントにメンバーシップを付与」として処理されます（エラーにはなりません）。

### 既存アカウントへの権限付与

**場所:** テナント詳細 → 「メンバー」タブ → 「権限付与・変更」フォーム

登録済みのメールアドレスとロールを入力して保存するだけです。招待メールは送信されません。
`is_active` のチェックを外すことでメンバーシップを無効化できます（削除はできません）。

### テナント作成時のオーナー設定

オーナーメールアドレスは **Supabase に登録済みである必要があります**。未登録の場合の手順:

1. 先に「招待メール送信」でアカウントを作成してもらう
2. 受信者がパスワードを設定してアカウントを有効化
3. その後、テナント作成フォームでメールアドレスを指定する

### 管理者アカウントの管理

管理コンソールへのログインが可能な管理者アカウントは、Supabase Dashboard の `platform_admin_users` テーブルで直接管理します。

| カラム | 説明 |
|--------|------|
| `user_id` | Supabase Auth の user ID |
| `role` | `owner` または `staff` |
| `is_active` | false にするとログイン不可 |

## 5. プランリミット管理

**パス:** `/sub-domain/plans`（サイドバーの「プラン管理」）
**権限:** `owner` のみ編集可能（`staff` は閲覧のみ）

### リミット項目一覧

| カラム名 | 説明 | 単位 |
|----------|------|------|
| `max_bots` | 作成できるBotの上限数 | 件 |
| `max_monthly_messages` | 月間チャットメッセージの上限数。超過すると応答が停止する。 | 件/月 |
| `max_storage_mb` | ソースファイルのストレージ上限 | MB |
| `max_hosted_pages` | Hosted URL として公開できるBotの上限数 | 件 |
| `internal_max_bots_cap` | `max_bots` に対する追加の上限キャップ（0 = 無効）。個別プラン調整に使用。 | 件 |
| `has_api` | APIキーによるチャットAPI利用の可否 | true/false |
| `has_hosted_page` | Hosted URL 公開機能の利用可否 | true/false |

**`internal_max_bots_cap` の補足:**
`max_bots` と `internal_max_bots_cap` が両方設定されている場合、小さい方の値が実際の上限になります。

```
実際のBot上限 = min(max_bots, internal_max_bots_cap)  ※ cap が 0 の場合は max_bots のみ
```

### 変更手順

1. 「プラン管理」ページを開く
2. 変更したいプランのカードで数値を編集
3. 「保存する」→「本当に更新する」の2段階確認で保存

### 変更時の影響範囲

| 操作 | 影響 |
|------|------|
| `max_bots` を減らす | 既存のBotは削除されない。新規作成のみブロックされる。 |
| `max_monthly_messages` を減らす | 月間使用量がすでに上限を超えている場合、即座に応答が停止する。 |
| `has_api` を false にする | APIキーでのチャットリクエストが即座に拒否される。 |
| `has_hosted_page` を false にする | Hosted URL ページへのアクセスが拒否される。 |

> 🚨 **プランリミットの変更はそのプランを契約している全テナントに即時反映されます。**
> 特定テナントだけ制限を変えたい場合は「手動Override」を使用してください。

## 6. 緊急停止手順

### 停止の種類

| 種類 | 対象 | 影響範囲 |
|------|------|----------|
| テナント強制停止 | テナント全体 | そのテナントの全Botの応答がブロック |
| Bot個別停止 | 特定のBot1件 | 対象Botのみ応答ブロック。他のBotは影響なし |
| テナント無効化 | テナント全体 | 解約済み扱い。プラン判定が無効になる |

**`force_stop` と `active = false` の違い:**

| | force_stop = true | active = false |
|---|---|---|
| チャット応答 | ブロック | プラン判定が無効扱い |
| コンソールログイン | 可能 | 可能 |
| データ | 保持 | 保持 |
| 用途 | 緊急停止 | 通常の解約処理 |

緊急時は原則 `force_stop` を使用してください。

### テナント全体を緊急停止する

**場所:** テナント詳細 → 「概要」タブ → 「テナント強制停止」カード

1. 停止理由を入力（任意。後から参照できるため記入推奨）
2. 「tenantを強制停止する」にチェック
3. 「停止状態を更新」→ 確認ダイアログで確定
4. テナントヘッダーに「強制停止中」バッジが表示されることを確認

**停止後の状態:** チャット API が `503` を返す / インデックスジョブが停止 / データは保持

### 特定の Bot だけを停止する

**場所:** テナント詳細 → 「Bot管理」タブ

対象Botの「強制停止する」にチェック → 「更新」ボタン → 確認ダイアログで確定

### 停止の解除

同じ画面でチェックを外して更新するだけです。即時反映されます。

### 緊急対応のエスカレーションフロー

```
1. 問題発覚
   ↓
2. 影響範囲を確認（Bot単体 or テナント全体）
   ↓
3. 該当範囲を強制停止（対応時間: 1〜2分）
   ↓
4. 原因調査（監査ログ・インデックスジョブ・Supabase Dashboardを確認）
   ↓
5. 問題が解決したら停止を解除
   ↓
6. 必要に応じてテナントに状況を連絡
```

### よくある緊急ケースと対応

| ケース | 推奨対応 |
|--------|----------|
| 大量の不正リクエスト | テナント強制停止 → 原因調査 |
| Botの回答内容が不適切 | 対象Botを個別停止 → 代理閲覧でBot設定を確認 |
| 支払い未収が続く | Stripe確認 → Overrideで `status: unpaid` に変更 → 必要に応じてテナント無効化 |
| テナントから退会申請 | テナント無効化（active = false）+ Stripe解約 |

### 停止操作の監査ログアクション

| 操作 | アクション |
|------|-----------|
| テナント強制停止 有効化 | `platform.tenant.force_stop.enable` |
| テナント強制停止 解除 | `platform.tenant.force_stop.disable` |
| Bot強制停止 有効化 | `platform.bot.force_stop.enable` |
| Bot強制停止 解除 | `platform.bot.force_stop.disable` |

## 7. 監視・ログ確認

### 監査ログ

**パス:** `/sub-domain/audit-logs`

コンソール上のすべてのミューテーション操作が記録されます。1ページ50件表示。

**アクション種別**

| プレフィックス | 説明 |
|----------------|------|
| `platform.*` | 管理コンソールでの操作（テナント管理・Override設定・代理閲覧など） |
| `console.*` | 契約者コンソールでの操作（Bot作成・設定変更など） |
| `bot.*` | Botに関する操作 |
| `source.*` | ソース（URL/PDF）の追加・削除 |
| `billing.*` | 請求・プラン関連の操作 |

**フィルタ機能:**
- tenant_id（UUID）入力で特定テナントの操作のみ表示
- アクション種別（`platform.*` など）でカテゴリ絞り込み

### インデックスジョブ監視

**パス:** `/sub-domain/indexing-jobs`

テナントが URL や PDF を登録するとインデックスジョブがキューに積まれます。RAG 検索の品質に直結するため、エラーや滞留がないか定期的に確認してください。

**ステータスの意味**

| status | 意味 |
|--------|------|
| `queued` | 実行待ち。通常は数秒〜数十秒で `running` になる。 |
| `running` | 現在処理中。 |
| `completed` | 正常完了。 |
| `failed` | エラーで失敗。`error_message` 列に詳細が記録されている。 |

**異常の判断基準**

| カード | 警告の目安 |
|--------|-----------|
| `queued` | 10件超で滞留の可能性 |
| `failed` | 1件以上で要調査 |
| `running` | 30分以上変化しない場合は異常の可能性 |

**failed 発生時の原因と対処**

| エラーパターン | 原因 | 対処 |
|----------------|------|------|
| `OpenAI API error` | OpenAI の障害またはレート制限 | 時間を置いて再試行 |
| `timeout` / `fetch failed` | 対象URLが応答しない | テナントにURL確認を依頼 |
| `storage quota exceeded` | ストレージ上限超過 | プランの `max_storage_mb` を確認 |
| `PDF parse error` | PDFが壊れている・暗号化されている | テナントにファイル確認を依頼 |

### 定期確認の推奨スケジュール

| 頻度 | 確認内容 |
|------|----------|
| 毎日 | インデックスジョブの `failed` 件数・`queued` 滞留確認 |
| 週次 | 監査ログで不審な操作がないか確認 |
| 月次 | プラン別メッセージ消費量・テナント数の推移確認（Supabase Dashboard） |

### Supabase Dashboard での詳細確認

管理コンソールでは直近データのみ表示されます。古い履歴が必要な場合は Supabase Dashboard を使用してください。

| テーブル | 用途 |
|----------|------|
| `audit_logs` | 全監査ログ（管理コンソールは最新50件/ページ） |
| `indexing_jobs` | 全インデックスジョブ（管理コンソールは直近100件） |
| `usage_daily` | テナント別の日次使用量 |
| `chat_logs` | チャット履歴の詳細 |
